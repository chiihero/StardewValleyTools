# Mod 更新功能实施方案

## 1. 目标

在当前 Python 版 Stardew Valley Mod Manager 中增加“Mod 更新检查”能力，先实现**检查、提示、缓存**，并补上**下载与安装**的更新链路。

## 2. 参考 Stardrop 的更新思路

Stardrop 的做法不是在界面里硬编码“某个 Mod 是否过期”，而是走一条完整链路：

1. 从 `manifest.json` 和更新键中提取可查询信息；
2. 通过更新接口获取远端建议版本；
3. 把结果写入缓存；
4. 启动时优先读缓存，手动刷新时再发起新的请求；
5. 把结果回填到 Mod 对象，再由 UI 展示。

对应到 Stardrop 里的核心落点是：

- `Models/Mod.cs`：保存 `SuggestedVersion`、`UpdateUri`，并负责版本比较；
- `Models/Data/UpdateCache.cs` / `ModUpdateInfo.cs`：缓存更新结果；
- `Utilities/External/SMAPI.cs`：批量查询 `smapi.io` 的更新数据；
- `Views/MainWindow.axaml.cs`：启动时读取缓存、手动检查更新、回写 UI。

## 3. 当前项目现状

当前项目已经具备这些基础：

- `src/models.py`：已有 `ManagedMod`、`ModAnalysis`、`WorkerEvent`；
- `src/scanner.py`：能解析 `manifest.json` 和 `i18n`；
- `src/storage.py`：已有本地状态文件 `.stardewvalleytools/state.json`；
- `src/ui.py`：已有主界面、后台线程、日志面板、进度区；
- `src/manager.py`：负责扫描与导入。

也就是说，这个功能最适合做成**独立更新服务 + 状态缓存 + UI 按钮入口**，不要把逻辑塞进扫描函数里。

## 4. 第一版范围（建议）

第一版只做以下内容：

1. 扫描时识别 Mod 的更新线索；
2. 查询远端更新数据；
3. 在列表中标记“有更新 / 已最新 / 无法判断 / 查询失败”；
4. 把最近一次检查结果写入本地状态；
5. 提供手动刷新按钮；
6. 对可更新 Mod 提供下载入口。

暂不做：

- 自动静默安装；
- 自动删除旧文件；
- 多来源聚合策略；
- 自更新程序本体。

## 5. 建议的模块拆分

### 5.1 `src/models.py`

新增更新相关数据结构，例如：

- `ModUpdateInfo`：保存 `unique_id`、`suggested_version`、`update_url`、`status`；
- `UpdateCache`：保存检查时间和 Mod 更新列表；
- `UpdateStatus`：统一表示 `up_to_date / outdated / unknown / failed`。

### 5.2 `src/scanner.py`

补充更新检查所需字段提取：

- `manifest.json` 里的 `uniqueID`、`version`；
- 若存在更新键，保留原始更新线索；
- 如果后续要支持 Stardrop 那种“按平台查更新”，这里要把线索标准化。

### 5.3 新增 `src/update_service.py`

这是核心服务层，负责：

- 汇总可查询的 Mod；
- 调用更新接口；
- 做版本比较；
- 生成缓存对象；
- 对外返回可供 UI 展示的结果。

如果要支持下载更新，这一层还需要负责：

- 根据更新源解析下载地址；
- 发起下载请求并保存到临时目录；
- 在下载完成后返回文件路径和元信息，供安装流程使用。

### 5.4 `src/storage.py`

扩展现有 `state.json`：

- 保存最近一次更新检查时间；
- 保存每个 Mod 的更新状态；
- 支持缓存读写与失效判断。

### 5.5 `src/ui.py`

新增 UI 入口：

- “检查 Mod 更新”按钮；
- 列表里新增更新状态列或详情字段；
- 检查中显示进度，失败时显示简短原因；
- 结果刷新后同步写回本地状态。

若加入下载能力，还需要增加：

- 单个 Mod 的“下载更新”按钮；
- 批量下载入口；
- 下载进度、完成提示和错误提示；
- 下载完成后是否立即安装的确认弹窗。

## 6. 数据流设计

建议按下面顺序跑：

1. `scan_mod()` 读取本地 Mod 信息；
2. `update_service` 提取可更新目标；
3. 优先读取缓存，缓存过期再发起请求；
4. 对远端版本和本地版本做比较；
5. 把结果写回 `ManagedMod`；
6. 更新 UI 列表与状态栏；
7. 落盘到 `state.json`。

下载更新时，再补一条分支：

1. 从更新结果中拿到下载链接；
2. 下载到本地临时目录；
3. 解析压缩包或目录结构；
4. 按安装策略覆盖或跳过已有文件；
5. 必要时先备份原 Mod；
6. 安装完成后重新扫描并刷新状态。

## 7. 参考 Stardrop 时要保留的思路

可以借鉴的点：

- **缓存优先**：启动后先用旧结果兜底；
- **异步检查**：避免阻塞 UI；
- **版本比较独立**：不要把比较逻辑塞进界面；
- **结果回写对象**：让 UI 只负责展示。
- **下载与安装分离**：先拿到可下载文件，再进入安装流程；
- **安装前确认**：避免误覆盖用户已有内容。

不建议照搬的点：

- Stardrop 的 Avalonia / ReactiveUI 界面组织；
- 它的 SMAPI/Nexus 深度安装流程；
- 其独立缓存目录结构。

## 8. 风险与约束

1. **更新来源不统一**：不同 Mod 可能来自 SMAPI、Nexus、GitHub 或其他来源；
2. **manifest 信息不完整**：没有更新键的 Mod 只能标记为“无法判断”；
3. **版本号格式不一致**：要容忍 `v1.2.3`、`1.2.3-beta` 之类格式；
4. **网络失败**：失败后只能回退到缓存或显示未知；
5. **缓存失效**：缓存要有时间戳，不能永久复用；
6. **下载包格式不统一**：有些 Mod 是 zip、7z、rar，甚至是页面链接而非直链。

## 9. 推荐实施顺序

### Phase 1：只做检查

- 定义更新数据模型；
- 提取 Mod 更新线索；
- 接入远端查询；
- 写入缓存；
- UI 展示状态。

### Phase 2：补充交互

- 支持单个 Mod 手动刷新；
- 支持批量刷新；
- 支持按“有更新”筛选。

### Phase 3：再考虑下载 / 安装

- 如果后续确认需要，可再补自动下载；
- 只建议先支持一种更新源；
- 下载后先校验文件，再进入安装；
- 安装前必须有确认弹窗和失败回退；
- 有覆盖风险时先提示用户是否备份原 Mod。

## 10. 验证标准

- 能扫描出带更新线索的 Mod；
- 能正确识别“有更新 / 无更新 / 未知”；
- 断网时不会让界面崩溃；
- 缓存能读写且不会覆盖掉现有 Mod 管理信息；
- 下载得到的安装包路径正确、可复用；
- 安装后能重新扫描并反映最新版本状态；
- UI 刷新后状态和本地数据一致。

## 11. 结论

最稳妥的落地方式是：**参考 Stardrop 的“缓存 + 异步检查 + 回填 UI”模式，但按当前 Python/tkinter 架构重做**。
建议第一版就把“检查 + 下载 + 安装入口”一起规划好，但实现时仍按“先检查，再下载，最后安装”的顺序逐步落地。
