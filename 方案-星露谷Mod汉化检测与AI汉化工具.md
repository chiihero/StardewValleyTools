# 星露谷物语 Mod 汉化检测与 AI 汉化工具方案

## 1. 目标

在当前目录新增一个**简单桌面工具**，用于：

1. 选择一个星露谷物语 Mod 目录；
2. 判断该 Mod 是否**已有汉化**；
3. 在未汉化或汉化不完整时，提供一个按钮触发 **AI 汉化**；
4. 生成汉化结果文件，尽量不覆盖原文件，便于人工复查。

当前工作区现状：根目录只有 `mod参考/`，其中包含 Content Patcher、Alternative Textures、Fashion Sense 等参考 Mod，没有现成 Python 应用代码，因此本项目按**绿地桌面工具**设计。

---

## 2. 首版范围（只做简单可用版）

### 包含

- 桌面 GUI；
- 选择本地 Mod 文件夹；
- 读取 `manifest.json` 判断是否为可识别的 Stardew/SMAPI Mod；
- 检测是否存在中文汉化；
- 显示检测结果；
- 点击按钮调用 AI 进行汉化；
- 将 AI 结果写入标准位置 `i18n/zh.json`；
- 保留日志与错误提示。
- 批量扫描整个 Mods 目录；

### 不包含

- 一键修复所有生态的特殊格式；
- 自动上传/发布汉化；
- 云端账号体系；
- 高级翻译记忆库；
- 覆盖所有历史老 Mod 的非标准结构。

这样能把首版控制在“简单工具软件”的范围内，先把主路径跑通。

---

## 3. 用户流程

1. 启动工具；
2. 点击“选择 Mod 文件夹”；
3. 工具读取目录并分析；
4. 界面显示：
   - 是否识别到 `manifest.json`；
   - Mod 类型；
   - 是否已有汉化；
   - 汉化完整度判断；
   - 可否执行 AI 汉化；
5. 若未汉化或汉化不完整，点击“AI 汉化”；
6. 工具提取待翻译文本，调用 AI；
7. 生成 `i18n/zh.json`；
8. 界面提示完成，并展示输出路径。

## 4. 技术选型

## 4.1 语言与 GUI

- **Python 3.11+**
- **tkinter** 作为 GUI

原因：

- 当前目录本来就是 PythonProjects；
- tkinter 为标准库，依赖最少，适合“简单工具”；
- 首版目标不是漂亮 UI，而是稳定、容易打包、容易维护。

## 4.2 AI 接入

- 使用官方 **OpenAI Python SDK**；
- 通过 `responses.create(...)` 调用模型；
- 要求模型输出严格 JSON；
- 在后台线程执行，避免 GUI 卡死。

## 4.3 文件处理

- 标准库：`pathlib`, `json`, `threading`, `queue`, `dataclasses`

## 5. Mod 汉化判定规则

首版按“**简单、保守、可解释**”原则判断，不做过度猜测。

## 5.1 识别 Mod 的基础条件

优先检查：

- 根目录是否存在 `manifest.json`

如果存在，则继续解析以下信息：

- `EntryDll`：通常表示 SMAPI 代码 Mod；
- `ContentPackFor`：通常表示内容包；
- `UniqueID` / `Name`：用于界面显示与日志。

若无 `manifest.json`，则提示“不是标准 SMAPI/内容包结构，暂不支持自动判断”。

## 5.2 已有汉化的判定

按以下顺序检测：

### A. 标准 `i18n` 文件结构

认为“存在中文汉化”的条件：

- `i18n/zh.json` 存在；或
- `i18n/zh/*.json` 存在。

这是首版最主要、最可靠的判断依据。

### B. 框架内嵌本地化字段

对部分生态，额外检查 JSON 中是否出现：

- `NameLocalization`
- `DescriptionLocalization`
- `TranslationKey`

说明：这类字段在部分 Mod 生态中可承载本地化信息，尤其是 Json Assets 一类。

### C. 判定结果分级

不要只给“有/没有”二值，建议界面分成：

- **已检测到中文汉化**：存在标准中文文件或明确中文本地化字段；
- **疑似部分汉化**：有本地化结构，但中文不完整；
- **未检测到汉化**：未发现标准中文资源；
- **无法判断**：结构不标准或生态暂未覆盖。

---

## 6. “汉化不完整”的简单判定

首版不做复杂语义分析，只做结构级比较：

### 标准 `i18n` 情况

若存在：

- `i18n/default.json`
- 以及 `i18n/zh.json`

则比较 key：

- `zh.json` 覆盖全部 key → 视为“较完整”；
- `zh.json` 缺少部分 key → 视为“部分汉化”；
- `zh.json` 不存在 → 视为“未汉化”。

### 子目录 `i18n/zh/` 情况

如果存在目录式结构，则按文件名与 key 逐级对比默认语言文件。

### 限制

- 不判断翻译质量；
- 不判断是否机器翻译；
- 不判断图片类文字资源是否已汉化；
- 不判断代码逻辑里动态生成的文本。

---

## 7. AI 汉化功能设计

## 7.1 触发条件

满足以下任一条件时启用按钮：

- 未检测到中文汉化；
- 检测到仅部分汉化；
- 用户主动选择“重新生成汉化”。

## 7.2 输入来源

优先级：

1. `i18n/default.json`
2. 其他可识别的默认语言资源
3. 框架 JSON 中待翻译字段

首版建议只把 **JSON 文本资源** 纳入 AI 汉化，避免直接处理图片、二进制、地图等复杂资产。

## 7.3 AI 提示词原则

必须要求模型：

- 输出为合法 JSON；
- 保持 key 不变；
- 保持占位符不变，如 `{{token}}`、`{0}`、`[SMAPI]`；
- 不擅自删除字段；
- 不新增注释；
- 语气适合游戏文本。

## 7.4 输出位置

标准输出：

- `i18n/zh.json`

若目标文件已存在，非强制模式下先检查是否缺少内容；完整则跳过，缺失则更新 `i18n/zh.json`。

强制模式允许直接覆盖 `i18n/zh.json`。

## 8. 推荐的首版架构

建议目录结构：

```text
StardewValleyTools/
  mod参考/
  app.py
  requirements.txt
  README.md
  src/
    ui.py
    scanner.py
    detector.py
    translator.py
    writers.py
    models.py
    prompts.py
```

模块职责：

- `app.py`：程序入口；
- `ui.py`：tkinter 窗口、按钮、状态展示；
- `scanner.py`：扫描目录、读取 manifest、定位资源；
- `detector.py`：汉化存在性与完整度判定；
- `translator.py`：调用 OpenAI；
- `writers.py`：备份并写出汉化文件；
- `models.py`：数据结构定义；
- `prompts.py`：AI 提示词模板。

这是足够清晰但不过度设计的拆分。

## 9. 数据结构建议

建议内部统一输出一个分析结果对象，例如：

- `mod_path`
- `mod_name`
- `mod_type`
- `has_manifest`
- `has_chinese`
- `translation_status`
- `default_locale_path`
- `zh_locale_path`
- `missing_keys_count`
- `translatable_sources`
- `warnings`

其中 `translation_status` 可取：

- `translated`
- `partial`
- `not_translated`
- `unknown`

这样 GUI 与后续 AI 流程都可复用同一份结果。

## 10. GUI 最小界面方案

首版界面只保留必要元素：

- 一个“选择 Mod 文件夹”按钮；
- 一个路径显示框；
- 一个“开始检测”按钮；
- 一个结果展示区域；
- 一个“AI 汉化”按钮；
- 一个日志/状态文本框。

结果区建议展示：

- Mod 名称
- Mod 类型
- 是否检测到中文
- 默认语言文件路径
- 中文文件路径
- 缺失 key 数量
- 建议操作

不做多页签，不做复杂表格，保持简单。

---

## 11. 错误处理与安全边界

需要明确提示的情况：

- 目录不存在；
- `manifest.json` 解析失败；
- `default.json` 不是合法 JSON；
- AI 接口密钥缺失；
- AI 返回结果不是合法 JSON；
- 输出文件写入失败。

安全原则：

- 不直接覆盖原始默认语言文件；
- 默认不覆盖已有正式 `zh.json`；
- 任何写操作都要先显示目标路径；
- AI 结果写入前必须经过 JSON 校验。

---

## 12. 验证标准（Acceptance Criteria）

实现阶段应以以下二元条件验收：

1. 选择一个带 `manifest.json` 的 Mod 目录后，工具能成功识别并显示基础信息；
2. 若存在 `i18n/zh.json`，工具显示“已检测到中文汉化”；
3. 若仅有 `i18n/default.json` 且没有中文文件，工具显示“未检测到汉化”；
4. 若 `zh.json` 缺少部分 key，工具显示“部分汉化”并给出缺失数量；
5. 点击“AI 汉化”后，程序不会阻塞 GUI；
6. AI 返回合法 JSON 时，工具能成功生成输出文件；
7. 输出文件内容保持原 key，不破坏占位符；
8. 出错时界面能展示明确错误，而不是静默失败。

---

## 13. 首版实现顺序

建议按下面顺序推进：

### 第一步：检测能力

- 先做目录选择；
- 读取 `manifest.json`；
- 检测 `i18n/default.json`、`i18n/zh.json`；
- 输出状态结果。

### 第二步：完整度比较

- 对比 default 与 zh 的 key；
- 显示缺失数。

### 第三步：AI 汉化

- 接入 OpenAI SDK；
- 翻译缺失项或整份默认语言；
- 写出到生成文件。

### 第四步：稳定性收尾

- 加日志；
- 加错误提示；
- 加简单 README。

## 15. 结论

这个工具的首版完全可以做成一个**基于 Python + tkinter 的轻量桌面应用**。核心判定规则以 `manifest.json` 和 `i18n/zh.json` / `i18n/default.json` 为主，再对部分生态补充本地化字段检测；AI 汉化只处理 JSON 文本资源，并将结果安全写出到新文件，避免误覆盖。

这条方案的优点是：范围小、实现快、可解释、便于后续逐步扩展到批量扫描和更多 Mod 生态。
